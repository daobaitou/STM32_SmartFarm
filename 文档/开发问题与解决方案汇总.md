## 问题1：DS18B20读取失败 - HAL_GPIO_Init切换模式过慢

### 现象
DS18B20单总线通信失败，读取温度始终返回0或85°C（默认值）。

### 原因
DS18B20使用Dallas 1-Wire协议，对时序要求极高（微秒级）。HAL库的`HAL_GPIO_Init()`函数内部执行了大量判断和配置操作，单次调用耗时数十微秒，严重超出时序窗口：
- 复位脉冲要求：拉低480μs → 释放 → 等待60μs → 检测存在脉冲
- 读写时隙要求：拉低1μs → 15μs内采样
- `HAL_GPIO_Init()`一次调用就耗掉了整个时序窗口

### 解决方案
绕过HAL库，直接操作GPIO寄存器（CRL）切换模式：

```c
#define DS18B20_PIN  GPIO_PIN_6
#define DS18B20_PORT GPIOA

/* 推挽输出50MHz: CNF=00, MODE=11 → 0x3 */
#define MODE_OUT() do { GPIOA->CRL = (GPIOA->CRL & ~0x0F000000) | 0x03000000; } while(0)
/* 上拉输入: CNF=10, MODE=00 → 0x8 */
#define MODE_IN()  do { GPIOA->CRL = (GPIOA->CRL & ~0x0F000000) | 0x08000000; } while(0)
```

使用DWT周期计数器实现精确微秒延时：
```c
#define DELAY_US(us) do { \
    uint32_t start = DWT->CYCCNT; \
    while ((DWT->CYCCNT - start) < (us * (SystemCoreClock / 1000000))); \
} while(0)
```

### 结果
- 温度读取稳定：25.0~25.1°C（室温）
- 重写了6次以上才找到根本原因

### 经验教训
- 单总线等时序敏感协议，不能用HAL_GPIO_Init切换模式
- 直接寄存器操作速度快（1~2个时钟周期），适合微秒级时序控制

## 问题2：DHT22读取校验失败 - 握手时序不完整

### 现象
DHT22能返回数据，但校验和始终不通过，温湿度值异常。

### 原因
DHT22通信协议的握手阶段包含4个步骤，原代码只实现了3个：

```
主机拉低 ≥1ms → 主机释放 → 从机拉低80μs → 从机拉高80μs
   (Step 1)      (Step 2)    (Step 3)        (Step 4) ← 缺失！
```

缺少第4步（等待从机拉高80μs结束），导致数据采样起点偏移，所有位数据错位，校验和不通过。

### 解决方案
在发送起始信号后，增加第4个等待循环：

```c
/* Step 1: 主机拉低 ≥1ms */
MODE_OUT();
HAL_GPIO_WritePin(DHT22_PORT, DHT22_PIN, GPIO_PIN_RESET);
DELAY_US(1200);

/* Step 2: 主机释放，切换输入 */
MODE_IN();
DELAY_US(30);

/* Step 3: 等待从机拉低响应 */
while (HAL_GPIO_ReadPin(DHT22_PORT, DHT22_PIN) == GPIO_PIN_SET);

/* Step 4: 等待从机拉低80μs结束 */
while (HAL_GPIO_ReadPin(DHT22_PORT, DHT22_PIN) == GPIO_PIN_RESET);

/* Step 5: 等待从机拉高80μs结束（关键！原代码缺失） */
while (HAL_GPIO_ReadPin(DHT22_PORT, DHT22_PIN) == GPIO_PIN_SET);
```

### 结果
- 温湿度读取正常，校验和通过
- 偶尔仍有读取失败（DHT22特性），增加重试机制解决

### 经验教训
- DHT22数据手册的握手时序图必须严格逐步实现
- 单总线设备的时序容差很小，每个等待步骤都不能省略

## 问题3：DHT22偶发读取失败

### 现象
DHT22大部分时间正常，但偶尔读取失败返回错误。

### 原因
- DHT22最小采样间隔2秒，读取过快会失败
- 单总线通信本身有概率出错（电磁干扰、线缆长度等）

### 解决方案
在读取函数中增加重试机制：

```c
HAL_StatusTypeDef DHT22_Read(DHT22_Data_t *data)
{
    for (uint8_t retry = 0; retry < 3; retry++)
    {
        if (dht22_read_raw(data) == HAL_OK)
            return HAL_OK;
        HAL_Delay(100);  // 等待后重试
    }
    return HAL_ERROR;
}
```

### 结果
- 读取成功率接近100%
- 偶尔的重试对系统影响可忽略

## 问题4：FC28土壤湿度ADC读数波动

### 现象
FC28传感器ADC值波动较大（±100~200），不在土壤中时ADC值约4095。

### 原因
- FC28是电阻式传感器，本身精度有限（±10%）
- 在空气中无负载，ADC值接近满量程（4095），属正常现象
- 放入土壤后ADC值会降至1200~2500范围

### 解决方案
无需修改驱动，这属于传感器物理特性。ADC值到湿度百分比的映射已做好线性转换：

```c
/* 干燥(高ADC)~0%, 潮湿(低ADC)~100% */
if (data->adc_value > 3500)
    data->moisture = 0;
else if (data->adc_value < 1200)
    data->moisture = 100;
else
    data->moisture = (uint8_t)((3500 - data->adc_value) * 100 / (3500 - 1200));
```

### 结果
- 在空气中显示湿度约0%（正确）
- 插入土壤后湿度值正常上升

### 经验教训
- 电阻式传感器波动是正常现象，不必追求完美稳定
- 实际使用中可通过软件滤波（滑动平均）进一步平滑数据

## 问题5：OLED编译失败 - 字体数组数据损坏

### 现象
编译报错：
```
display_oled.c(37): error: #146: too many initializer values
    {0x3E,0x51,0x49,0x45,0x45,0x45,0x39}, /* 0 */
display_oled.c(41): error: #146: too many initializer values
    {0x00,0x18,0x14,0x12,0x7F,0x14,0x7F,0x14,0x00}, /* # */
```

### 原因
font6x8字体数组中存在两处数据损坏：
1. 字符'0'的数组有7个字节（应为6个）
2. 字符'#'的数组有9个字节（应为6个）
3. 数组中存在重复的字符定义（space到'/'的区间被重复定义了两次）

### 解决方案
使用标准6x8 ASCII字体表替换整个font6x8数组，确保95个字符（0x20~0x7E）每个恰好6字节。同时添加了缺失的`#include <string.h>`（memset需要）。

### 经验教训
- 字体数据建议从可靠来源获取，手动编辑容易出错
- 编译器的"too many initializer values"错误直接指向数据定义问题

## 问题6：OLED烧录后黑屏不显示

### 现象
代码编译通过，烧录成功，但OLED屏幕完全黑屏，无任何显示。

### 排查过程
1. 添加I2C设备扫描 → `OLED_Init()`返回HAL_OK，说明I2C通信本身成功
2. 添加`OLED_Fill(0xFF)`全屏点亮测试 → OLED正常显示所有像素
3. 结论：初始化和I2C通信正常，问题可能出在后续的显示内容

### 原因
原始测试代码的显示逻辑在添加诊断代码时被覆盖，导致主循环显示内容丢失。

### 解决方案
恢复正常的显示流程后，OLED显示正常。

### 经验教训
- 调试显示问题时，先用全屏填充（0xFF）测试硬件是否正常
- 分层排查：硬件连接 → I2C通信 → 初始化序列 → 显示内容

## 问题7：ST-Link无法进入调试模式

### 现象
Keil烧录时报错：
```
CMSIS-DAP - Cortex-M Error: Cannot enter Debug Mode
```

### 原因
芯片可能处于I2C死锁状态，SWD调试接口被占用或芯片跑飞。

### 解决方案
**按住Reset键 → 点击Keil下载 → 松开Reset键**

这样ST-Link在芯片复位瞬间接管调试接口，避免被异常状态阻塞。

### 经验教训
- 当芯片处于异常状态（死锁、跑飞）时，常规下载会失败
- "按住Reset→下载→松开"是通用解决方案

## 问题8：OLED + BH1750共用I2C总线导致系统死机（最严重）

### 现象
OLED和BH1750共用I2C1总线时，系统运行10~25秒后完全停止：
- 串口停止输出
- OLED停止刷新
- LED停止闪烁
- 必须手动复位恢复

### 根本原因（三个因素叠加）

**因素1：OLED逐字节发送，I2C事务过多**
```c
// 原代码：每页128字节逐个发送 = 1024次I2C事务/每帧
for (uint8_t col = 0; col < 128; col++)
    oled_write_data(oled_buf[page][col]);  // 每字节一次完整I2C事务
```

**因素2：BH1750连续模式占用总线**
```c
uint8_t cmd = 0x10;  // 连续测量模式，持续占用总线
```

**因素3：STM32F103 I2C硬件缺陷**
- I2C总线BUSY标志可能卡死，无法自动恢复
- 从设备拉低SDA后主设备无法发送STOP信号
- 一旦锁死，所有后续I2C操作永久阻塞

### 解决方案（四步）

**步骤1：OLED批量发送（1024次 → 8次）**
```c
static uint8_t page_buf[1 + 128];
page_buf[0] = 0x40;  // data control byte
memcpy(&page_buf[1], oled_buf[page], 128);
HAL_I2C_Master_Transmit(&hi2c1, OLED_ADDR << 1, page_buf, 129, 200);
```

**步骤2：BH1750改为单次触发模式**
```c
uint8_t cmd = 0x21;  // 一次高分辨率模式（非连续）
HAL_I2C_Master_Transmit(&hi2c1, BH1750_ADDR << 1, &cmd, 1, 100);
HAL_Delay(140);  // 等待测量完成
HAL_I2C_Master_Receive(&hi2c1, BH1750_ADDR << 1, buf, 2, 100);
```

**步骤3：I2C总线硬件恢复**
```c
static void i2c_reset(void)
{
    HAL_I2C_DeInit(&hi2c1);

    // 手动产生16个SCL脉冲释放从设备
    GPIO_InitTypeDef GPIO_InitStruct = {0};
    GPIO_InitStruct.Pin = GPIO_PIN_6;  // PB6 = I2C1_SCL
    GPIO_InitStruct.Mode = GPIO_MODE_OUTPUT_OD;
    GPIO_InitStruct.Speed = GPIO_SPEED_FREQ_HIGH;
    HAL_GPIO_Init(GPIOB, &GPIO_InitStruct);

    for (uint8_t i = 0; i < 16; i++)
    {
        HAL_GPIO_WritePin(GPIOB, GPIO_PIN_6, GPIO_PIN_RESET);
        HAL_Delay(1);
        HAL_GPIO_WritePin(GPIOB, GPIO_PIN_6, GPIO_PIN_SET);
        HAL_Delay(1);
    }

    HAL_I2C_Init(&hi2c1);  // 重新初始化I2C
}
```

**步骤4：失败自动重试**
```c
HAL_StatusTypeDef ret = HAL_I2C_Master_Transmit(...);
if (ret != HAL_OK)
{
    i2c_reset();                      // 恢复总线
    ret = HAL_I2C_Master_Transmit(...); // 重试一次
}
```

### 结果
- 系统稳定运行，不再卡死
- I2C偶发失败时自动恢复（用户无感知）
- OLED刷新效率提升128倍

### 后续优化建议
如果I2C问题仍然频繁出现，可考虑：
1. **GPIO模拟I2C** — 完全绕开硬件缺陷，最彻底的解决方案
2. **降低刷新频率** — 从1秒改为2秒，减少总线压力
3. **分离I2C总线** — OLED用I2C1，BH1750用I2C2

### 经验教训
- STM32F103的I2C硬件有已知缺陷，多设备共用时必须做错误恢复
- I2C批量传输比逐字节传输更高效也更稳定
- 单次触发模式比连续模式更适合多设备共享总线场景

## 问题9：GPIO模拟I2C替代硬件I2C

### 现象
问题8的HAL I2C恢复机制仍偶尔失败，系统有概率重启。

### 原因
STM32F103 I2C硬件缺陷是芯片硅片级bug（写在官方Errata中），无法通过软件补丁根治。恢复机制是"事后补救"，偶发情况下恢复不及时就导致芯片跑飞重启。

### 解决方案
编写GPIO模拟I2C驱动（soft_i2c），完全绕开硬件I2C外设：

```c
/* PB6=SCL, PB7=SDA, 开漏输出 */
#define SCL_H()  HAL_GPIO_WritePin(GPIOB, GPIO_PIN_6, GPIO_PIN_SET)
#define SCL_L()  HAL_GPIO_WritePin(GPIOB, GPIO_PIN_6, GPIO_PIN_RESET)
#define SDA_H()  HAL_GPIO_WritePin(GPIOB, GPIO_PIN_7, GPIO_PIN_SET)
#define SDA_L()  HAL_GPIO_WritePin(GPIOB, GPIO_PIN_7, GPIO_PIN_RESET)

/* 关中断保护每个字节的时序 */
void Soft_I2C_WriteByte(uint8_t data)
{
    uint32_t primask = __get_PRIMASK();
    __disable_irq();
    // ... 8位数据逐位发送 ...
    __set_PRIMASK(primask);
}
```

关键设计要点：
- 关中断保护START/STOP/字节传输时序，防止SysTick打断
- 50kHz时钟速率，兼容性好
- NACK时自动发送9个SCL脉冲 + STOP释放总线
- 每次操作最多重试3次

### 结果
- 系统不再卡死（硬件I2C锁死问题完全消除）
- I2C通信100%可控，不受硬件缺陷影响

## 问题10：BH1750光照值异常跳变导致系统重启

### 现象
系统稳定运行数百次循环后偶尔重启，串口日志显示重启前光照值剧烈跳变：
```
[83] Light: 5.8 lx
[84] Light: 245.8 lx    ← 异常跳变
[85] Light: 0.0 lx      ← 异常
[86] System Init OK!     ← 芯片重启
```

### 原因
BH1750在光照快速变化（如手遮挡/放开传感器）时，单次模式测量可能返回异常数据。虽然软件I2C本身不会锁死，但异常数据可能触发后续代码的意外行为。

### 解决方案
在BH1750驱动中增加数据合理性校验：

```c
/* 数据合理性校验: BH1750量程 0~65535lx */
if (light > 100000.0f)
    return HAL_ERROR;

/* 异常跳变过滤: 相邻两次变化超过10倍视为异常 */
if (last_light > 10.0f && light > last_light * 10.0f)
    return HAL_ERROR;
```

同时将测量等待时间从140ms增加到200ms，给传感器更充足的转换时间。

### 结果
- 重启概率大幅降低（从运行几十次循环必重启 → 运行数百次偶尔重启）
- 异常数据被过滤，不会影响系统判断

### 经验教训
- 传感器数据校验是系统健壮性的重要防线
- 模拟量/数字传感器都可能有偶发异常值，必须做合理性检查
- 物理量变化通常是连续的，突变超过合理范围即可判定为异常

## 问题11：FreeRTOS集成后任务完全不运行

### 现象
烧录成功，串口打印 "FreeRTOS starting..." 但后续没有任何任务输出：
- LED不闪烁
- 串口不打印传感器数据
- OLED不刷新

### 原因
SysTick被HAL_Init()默认配置为HAL tick，但SysTick_Handler已被改成调用xPortSysTickHandler()。在FreeRTOS调度器启动前，SysTick中断触发时FreeRTOS内核还没初始化，导致崩溃。

### 解决方案
重写HAL_InitTick()让HAL改用TIM4，SysTick完全留给FreeRTOS：

```c
/* tim.c - Override weak HAL_InitTick() */
HAL_StatusTypeDef HAL_InitTick(uint32_t TickPriority)
{
    __HAL_RCC_TIM4_CLK_ENABLE();

    htim4.Instance = TIM4;
    htim4.Init.Prescaler = 72 - 1;     /* 72MHz / 72 = 1MHz */
    htim4.Init.Period = 1000 - 1;      /* 1MHz / 1000 = 1kHz = 1ms */
    // ...

    HAL_TIM_Base_Start_IT(&htim4);
    HAL_NVIC_SetPriority(TIM4_IRQn, TickPriority, 0);
    HAL_NVIC_EnableIRQ(TIM4_IRQn);

    return HAL_OK;
}

/* stm32f1xx_it.c */
void SysTick_Handler(void) { xPortSysTickHandler(); }
void TIM4_IRQHandler(void) { HAL_TIM_IRQHandler(&htim4); }
void HAL_TIM_PeriodElapsedCallback(TIM_HandleTypeDef *htim) {
    if (htim->Instance == TIM4) HAL_IncTick();
}
```

### 结果
- FreeRTOS调度器正常启动
- 4个任务（LED/Sensor/LCD/Print）正常运行
- LED每500ms翻转，串口每2秒打印数据

### 经验教训
- FreeRTOS需要独占SysTick，HAL必须用其他定时器做tick
- 调度器启动前的SysTick中断会导致内核崩溃
- HAL_InitTick()重写必须在HAL_Init()之前生效

## 问题12：HAL TIM模块链接错误

### 现象

```
Error: L6218E: Undefined symbol HAL_TIM_IRQHandler
```

- 使用新的HAL模块必须同时做两件事：添加源文件 + 启用宏定义
- HAL条件编译会完全跳过未启用的模块

## 问题13：FreeRTOS任务栈溢出导致无输出

### 原因
LED任务栈太小（64字），加了printf后栈溢出。

### 解决方案

```c
xTaskCreate(vTask_LED, "LED", 128, NULL, PRIORITY_LED, NULL);  // 64→128
```

### 经验教训
- 使用printf的任务需要更大栈空间
- FreeRTOS栈单位是字（4字节），128字=512字节

## 问题14：FreeRTOS中断优先级配置错误导致Hard Fault

### 现象
系统启动后约3秒直接Hard Fault，无任何错误输出。

### 原因
`configMAX_SYSCALL_INTERRUPT_PRIORITY` 被设为 `11`，但STM32F103使用4位NVIC优先级（0~15），FreeRTOS要求该值按 `(优先级 << 4)` 格式编码：
- 期望优先级5 → 应编码为 `(5 << 4) = 80 (0x50)`
- 错误值11 → 被解析为优先级0（最高），导致FreeRTOS中断管理混乱

### 解决方案
```c
/* FreeRTOSConfig.h */
#define configMAX_SYSCALL_INTERRUPT_PRIORITY  (5 << 4)  /* 0x50, priority 5 */
```

### 经验教训
- STM32F103的NVIC优先级寄存器是8位，有效位左对齐到高4位，必须 `<< 4`
- 错误的中断优先级会导致FreeRTOS临界区保护失效，引发不可预测的Hard Fault

## 问题15：多任务栈溢出导致系统运行数秒后停止

### 现象
系统运行几秒后完全停止（串口无输出、LED不闪），板载LED常亮（栈溢出钩子触发）。

### 原因
多个任务栈空间不足，printf函数需要300+字节栈空间：
- LED任务：64字（256B）→ printf直接溢出
- Print任务：128字（512B）→ 勉强够用
- LCD任务：256字（1024B）→ OLED I2C + snprintf 紧张
- Sensor任务：384字（1536B）→ 6个传感器 + 多次printf 紧张

### 解决方案
```c
xTaskCreate(vTask_Sensor,  "Sensor",   512, NULL, PRIORITY_SENSOR,   NULL);  // 384→512
xTaskCreate(vTask_LCD,     "LCD",      384, NULL, PRIORITY_LCD,      NULL);  // 256→384
xTaskCreate(vTask_Print,   "Print",    192, NULL, PRIORITY_PRINT,    NULL);  // 128→192
xTaskCreate(vTask_LED,     "LED",      128, NULL, PRIORITY_LED,      NULL);  // 64→128
```

### 经验教训
- 使用printf的任务至少需要128字（512字节）栈
- 多次printf或snprintf格式化浮点数需要更多栈空间
- FreeRTOS `configCHECK_FOR_STACK_OVERFLOW=2` 可检测溢出，但钩子函数自身也要避免栈溢出

## 问题16：栈溢出钩子自身溢出无法输出任务名

### 现象
栈溢出钩子使用printf打印任务名，但钩子触发时任务栈已损坏，printf自身也溢出，导致二次崩溃，无法看到任何输出。

### 解决方案
改用直接操作USART1寄存器发送字符，几乎不消耗栈空间：

```c
void vApplicationStackOverflowHook(TaskHandle_t xTask, char *pcTaskName)
{
    const char *msg = "!!!OVF:";
    while(*msg) {
        while(!(USART1->SR & USART_SR_TXE));
        USART1->DR = *msg++;
    }
    if(pcTaskName) {
        char c;
        while((c = *pcTaskName++)) {
            while(!(USART1->SR & USART_SR_TXE));
            USART1->DR = c;
        }
    }
    while(1);
}
```

### 经验教训
- 栈溢出钩子运行在已溢出的任务上下文中，不能使用任何需要栈的函数（printf、malloc等）
- 直接写外设寄存器是最安全的调试输出方式

## 问题17：ESP-12E NodeMCU板载CH340与STM32信号冲突

### 现象
STM32通过USART2连接ESP-12E的GPIO1(TX)/GPIO3(RX)引脚，AT命令无响应或收到乱码。

### 原因
NodeMCU开发板板载CH340 USB-UART芯片**永久连接**到GPIO1/GPIO3，当STM32也驱动这两个引脚时：
- CH340的TX输出与STM32的PA2(TX)同时驱动GPIO1(RX) → 信号冲突
- CH340的RX输入会拉低GPIO3(TX) → STM32收到乱码

### 解决方案
刷入原始ESP-AT v2.2.2.0固件，该固件将AT UART映射到GPIO15(TX)/GPIO13(RX)（即NodeMCU的D8/D7引脚），避开CH340占用的GPIO1/GPIO3：

```
STM32 PA2 (TX) → ESP D7 (GPIO13/RX)
STM32 PA3 (RX) → ESP D8 (GPIO15/TX)
```

### 经验教训
- NodeMCU开发板的GPIO1/GPIO3被CH340占用，不能同时作为外部UART使用
- 不同ESP-AT固件版本的UART引脚映射不同，需查阅固件文档确认
- 使用ESP-01S等最小模块无此问题（无板载CH340）

## 问题18：ESP-AT固件MQTT命令不可用

### 现象
ESP-AT固件的MQTT查询命令（`AT+MQTT?`）返回正常，但所有MQTT设置命令（`AT+MQTTUSERCFG`、`AT+MQTTCONN`等）返回ERROR。

### 原因
固件MQTT SET功能异常，属于ESP-AT固件已知兼容性问题。

### 解决方案
绕过AT MQTT命令，使用原始TCP连接 + 手动编码MQTT 3.1.1协议数据包：

```c
/* 建立TCP连接到Broker */
AT+CIPSTART="TCP","124.223.5.91",1883

/* 手动构建MQTT CONNECT包 */
static uint16_t mqtt_build_connect(uint8_t *buf, const char *client_id)
{
    uint16_t pos = 0;
    buf[pos++] = 0x10;  // CONNECT packet type
    // ... 编码可变头 + payload ...
    return pos;
}

/* 通过TCP发送MQTT数据包 */
ESP8266_TCP_Send(mqtt_tx_buf, len);
```

支持的MQTT操作：
- CONNECT：连接Broker，等待CONNACK
- PUBLISH：发布传感器数据（JSON格式）
- SUBSCRIBE：订阅3个控制主题
- PINGREQ：保活心跳

### 经验教训
- 当AT命令不可用时，可通过原始TCP + 协议编码绕过
- MQTT 3.1.1协议相对简单，手动编码完全可行
- 需要处理MQTT Remaining Length可变长度编码

## 问题19：FreeRTOS队列多消费者问题

### 现象
LCD任务（优先级2）消耗了所有传感器队列数据，Print和MQTT_Pub任务收不到数据。

### 原因
FreeRTOS队列是单消费者模型——每条消息只能被一个任务取走。3个消费者共享1个长度为3的队列，高优先级任务取走所有消息，低优先级任务饿死。

### 解决方案
Sensor任务每次采集后向队列发送3份数据副本，每个消费者各取一份：

```c
/* 发送到队列（发送3份，每个消费者各取一份） */
xQueueSend(xQueue_SensorData, &data, 0);
xQueueSend(xQueue_SensorData, &data, 0);
xQueueSend(xQueue_SensorData, &data, 0);
```

同时将Print和MQTT_Pub优先级提升到2，与LCD同级，确保公平调度。

### 经验教训
- FreeRTOS队列是点对点传输，不支持广播
- 多消费者场景需要发送N份副本，或使用事件组/广播机制
- 优先级相同的任务通过时间片轮转公平调度

## 问题20：MQTT_Init阻塞FreeRTOS调度器启动

### 现象
在main()中调用MQTT_Init()，WiFi连接+MQTT握手耗时18+秒，期间所有FreeRTOS任务无法运行。

### 原因
MQTT_Init()在vTaskStartScheduler()之前调用，此时FreeRTOS调度器尚未启动，所有阻塞延时（HAL_Delay、vTaskDelay）都在空转，浪费启动时间。

### 解决方案
将MQTT初始化移入vTask_MQTT_Pub任务内部，先等5秒让ESP8266启动完成，再进行初始化：

```c
void vTask_MQTT_Pub(void *pvParameters)
{
    vTaskDelay(pdMS_TO_TICKS(5000));  // 等待ESP8266启动

    for (int retry = 0; retry < 10; retry++)
    {
        if (MQTT_Init(...) == HAL_OK) break;
        vTaskDelay(pdMS_TO_TICKS(5000));  // 重试间隔
    }
    // ...
}
```

### 经验教训
- 耗时操作不应放在main()中调度器启动前
- FreeRTOS任务中可使用vTaskDelay让出CPU，不影响其他任务运行
- 外设初始化放在对应任务中，职责更清晰

## 问题21：USART2多任务竞争导致AT命令失败

### 现象
MQTT_Pub和MQTT_Sub任务同时操作ESP8266，AT命令响应混乱或超时。

### 原因
两个任务共享USART2，无互斥保护：
- MQTT_Pub发送AT+CIPSEND时，MQTT_Sub读取了响应
- MQTT_Sub轮询+IPD时，MQTT_Pub的AT命令被截断

### 解决方案
添加xMutex_USART2互斥锁，保护所有USART2操作：

```c
SemaphoreHandle_t xMutex_USART2 = NULL;

static uint8_t take_mutex(uint32_t timeout_ms)
{
    return xSemaphoreTake(xMutex_USART2, pdMS_TO_TICKS(timeout_ms)) == pdTRUE;
}

/* 所有ESP8266操作前后加锁/解锁 */
AT_Result_t ESP8266_SendCmd(...)
{
    if (!take_mutex(timeout_ms + 1000)) return AT_TIMEOUT;
    // ... AT命令操作 ...
    give_mutex();
    return result;
}
```

### 经验教训
- 共享外设（UART、I2C、SPI）在多任务环境中必须加互斥锁
- 互斥锁超时不能太短，AT命令响应可能需要数秒
- 优先级继承（FreeRTOS互斥量自带）可避免优先级反转

## 问题22：YFS201水流传感器脉冲计数始终为零

### 现象
YFS201传感器读取始终返回0.00L/min，脉冲计数为0，EXTI0中断从未触发。

### 原因
硬件接线问题（待确认）：
- 传感器可能未正确供电（需5V）
- 信号线可能未接PB0
- 无水流时传感器不产生脉冲（正常）

### 当前状态
软件驱动正确（PB0 + EXTI0下降沿中断），需检查硬件接线：
- 红线 → 5V
- 黑线 → GND
- 信号线（黄/绿） → PB0

### 排查方法
- 对传感器吹气或拨动叶轮模拟脉冲
- 用杜邦线将PB0短接GND模拟下降沿，验证中断是否触发

## 问题23：DHT22读取始终失败

### 现象
DHT22读取始终返回"DHT22 failed"，温度和湿度值为-0.0。

### 原因
缺少上拉电阻。DHT22单总线需要4.7K上拉电阻连接到VCC，没有上拉电阻时信号线电平不稳定，无法完成握手。

### 解决方案
在DHT22数据线（PA1）和VCC之间添加4.7K上拉电阻。

### 经验教训
- 单总线设备必须加上拉电阻（DHT22内部上拉不够）
- 读取失败时先检查硬件（上拉电阻、接线），再排查软件时序

## 通用经验总结

| 类别 | 经验 |
|------|------|
| 时序敏感协议 | DS18B20/DHT22等单总线设备，不能用HAL_GPIO_Init，必须直接操作寄存器 |
| 精确延时 | 使用DWT周期计数器实现微秒延时，比定时器更精确 |
| I2C共享总线 | STM32F103硬件I2C有缺陷，推荐GPIO模拟I2C + 关中断保护时序 |
| FreeRTOS集成 | HAL必须用TIM替代SysTick，重写HAL_InitTick() |
| FreeRTOS中断 | configMAX_SYSCALL_INTERRUPT_PRIORITY必须按(优先级<<4)编码 |
| 任务栈大小 | printf任务需要≥128字（512字节），多传感器+格式化需要≥192字 |
| 栈溢出调试 | 钩子函数不能使用printf，必须直接写UART寄存器 |
| HAL模块启用 | 添加源文件 + 启用宏定义，两者缺一都会链接失败 |
| 调试方法 | 分层排查：硬件→通信→初始化→业务逻辑 |
| 传感器特性 | 电阻式传感器（FC28）波动属正常，不必过度优化 |
| 传感器上拉 | DHT22等单总线设备必须外接4.7K上拉电阻 |
| ESP8266接线 | NodeMCU板载CH340占用GPIO1/GPIO3，必须用其他引脚 |
| AT命令替代 | AT MQTT不可用时，可用原始TCP + 协议手动编码绕过 |
| 队列多消费者 | FreeRTOS队列不支持广播，需发送N份副本 |
| 共享外设保护 | 多任务访问同一UART/I2C必须加互斥锁 |
| ST-Link连接 | 无法下载时用"按住Reset→下载→松开"方法 |
| 数据校验 | 传感器数据必须有合理性校验（量程+变化率），异常值应丢弃 |
| 循环计数器 | 主循环加计数器打印，可快速判断是卡死还是重启 |
